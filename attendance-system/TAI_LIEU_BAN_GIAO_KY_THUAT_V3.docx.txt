Mục tiêu của Dự ánXây dựng hệ thống backend hiệu năng cao bằng ngôn ngữ Go (Golang) có khả năng kết nối song song và xử lý hàng triệu bản ghi chấm công từ hàng trăm thiết bị đầu cuối.Tích hợp thành công đa dạng giao thức kết nối phần cứng: zkemkeeper.dll COM SDK (ZKTeco Standalone), ADMS Push Protocol (ZKTeco tự đẩy dữ liệu qua HTTP), ISAPI REST (Hikvision) và REST API HTTP (Sunbeam).Thiết lập kho lưu trữ vân tay tập trung tại database PostgreSQL và xây dựng cơ chế tự động phân phối mẫu vân tay xuống các thiết bị đích, hỗ trợ quản lý tương thích phiên bản thuật toán vân tay.Tự động hóa hoàn toàn quy trình phê duyệt đơn từ (nghỉ phép, làm thêm giờ, sửa giờ công) và đồng bộ trực tiếp kết quả vào dữ liệu tính công ngày.Gia tăng trải nghiệm nhân viên thông qua các kênh thông báo tức thời (gửi email, tin nhắn Zalo OA khi quẹt thẻ) và cảnh báo nhắc nhở quên check-out tự động.1.2. SƠ ĐỒ LUỒNG DỮ LIỆU CHUYÊN SÂU2.1. Phân tích chi tiết Kiến trúc Clean Architecture trong GoHệ thống tuân thủ nghiêm ngặt nguyên lý thiết kế Clean Architecture của Robert C. Martin, phân chia mã nguồn thành 4 vòng tròn đồng tâm độc lập. Mục tiêu cốt lõi là làm cho logic nghiệp vụ hoàn toàn độc lập với database, UI framework và các thư viện kết nối phần cứng.Cấu trúc phân mục và vai trò chi tiết:Thư mục cmd/server: Chứa entrypoint main.go của ứng dụng. Nhiệm vụ chính là khởi tạo các kết nối cơ sở dữ liệu (PostgreSQL Connection Pool), đọc tham số cấu hình từ config.yaml hoặc biến môi trường, đăng ký dependency injection và khởi động server HTTP Gin cùng scheduler chạy nền.Thư mục internal/domain: Là trung tâm của kiến trúc (Core), định nghĩa các struct thực thể nghiệp vụ (như Device, Employee, AttendanceLog, Shift, DailyAttendance...) và định nghĩa các interface (Port) cho tầng lưu trữ (Repository) và tầng adapter phần cứng (DeviceAdapter). Tầng này độc lập hoàn toàn và không import bất kỳ thư viện ngoài nào (như gorm, sql, hay driver thiết bị).Thư mục internal/usecase: Chứa đựng toàn bộ các luật nghiệp vụ (Business Rules). Ví dụ: file attendance_processor_service.go chứa thuật toán phân tích log, tính giờ trễ/sớm và ghép cặp ca đêm; biometric_service.go điều phối việc đăng ký vân tay và đẩy lệnh hàng đợi xuống máy ADMS. Tầng này chỉ tương tác với các interface ở lớp Domain thông qua dependency injection.Thư mục internal/infrastructure: Chứa code chi tiết về công nghệ (Adapter). Bao gồm thư mục postgres/ (kết nối vật lý DB bằng pgxpool, thực thi các câu lệnh SQL), zkteco/ (triển khai COM SDK zkemkeeper.dll qua go-ole), hikvision/ (gọi API HTTP Digest REST của hãng Hikvision), sunbeam/ (REST API) và scheduler/ (quản lý cron ticker 10 giây/lần).Thư mục internal/interface/http: Tiếp nhận các request HTTP từ Web UI client. Bao gồm router cấu hình định tuyến API, handlers (nhận request DTO, kiểm tra hợp lệ đầu vào, chuyển tiếp xuống Usecase, định dạng output JSON trả về) và phân hệ SSE (Server-Sent Events) đẩy log real-time.2.2. Thiết kế trừu tượng hóa phần cứng (Hardware Abstraction Layer)Để giải quyết triệt để bài toán tích hợp đa chủng loại thiết bị và không phụ thuộc nhà cung cấp, hệ thống thiết kế lớp trừu tượng thông qua interface DeviceAdapter ở tầng Domain:type DeviceAdapter interface {&#10;    TestConnection(ctx context.Context, dev *entity.Device) error&#10;    GetAttendanceLogs(ctx context.Context, dev *entity.Device, fromTime time.Time) ([]entity.AttendanceLog, error)&#10;    SyncEmployee(ctx context.Context, dev *entity.Device, emp *entity.Employee) error&#10;    DeleteEmployee(ctx context.Context, dev *entity.Device, employeeCode string) error&#10;    EnrollFingerprint(ctx context.Context, dev *entity.Device, employeeCode string, fingerIndex int) error&#10;    ClearEnrollData(ctx context.Context, dev *entity.Device) error&#10;}Khi hệ thống khởi chạy hoặc nhận yêu cầu tương tác thiết bị, lớp Factory (DeviceAdapterFactory) sẽ đọc trường device_type của thiết bị lưu dưới DB để trả về instance tương ứng:zkteco (Standalone): Khởi tạo adapter kết nối COM SDK ActiveX (zkemkeeper.dll) chạy trên Windows qua cầu nối go-ole.zkteco (ADMS): Trả về adapter tương tác thông qua hàng đợi lệnh (device_command_queue) gửi phản hồi HTTP cho máy chấm công ADMS.hikvision: Trả về adapter gọi API ISAPI qua HTTP REST sử dụng Digest Authentication.sunbeam: Trả về adapter REST API chuẩn HTTP JSON.2.3. Sơ đồ Luồng Dữ liệu (Data Flow) Đồng bộ Dual-SyncĐể đảm bảo log chấm công được kéo về ngay lập tức nhưng vẫn có tính dự phòng và an toàn khi kết nối mạng chập chờn, hệ thống kết hợp 2 luồng đồng bộ chạy song song:Luồng 1: Push Protocol - Máy chấm công tự đẩy dữ liệu (ZKTeco ADMS)Bước 1: Nhân viên quẹt vân tay hoặc nhận diện khuôn mặt thành công trên máy chấm công ADMS.Bước 2: Thiết bị đóng gói bản ghi log và gửi request HTTP POST /iclock/cdata?SN=...&amp;table=ATTLOG lên Go server.Bước 3: Lớp Interface HTTP (ADMS Handler) tiếp nhận request, parse dữ liệu text thô của ADMS thành mảng thực thể AttendanceLog.Bước 4: Go server lưu bản ghi vào database Postgres. Nhờ cấu hình UNIQUE, log trùng lặp sẽ bị loại bỏ.Bước 5: Phát tín hiệu SSE đẩy log lên màn hình dashboard giám sát của admin.Bước 6: Spawn một Goroutine chạy nền gọi NotificationService để gửi tin nhắn thông báo tức thời cho nhân sự qua Zalo OA và Email.Bước 7: Thiết bị ADMS tiếp tục gọi HTTP GET /iclock/getrequest?SN=... để lấy các lệnh chờ trong hàng đợi (device_command_queue) như Enroll, Delete, Reboot... và thực thi.Luồng 2: Pull Protocol - Quét chủ động (Standalone SDK &amp; Hikvision/Sunbeam)Bước 1: Scheduler trong nhân hệ thống chạy nền định kỳ mỗi 10 giây.Bước 2: Scheduler gọi SyncService thực hiện lặp qua danh sách thiết bị có adms_enabled = false.Bước 3: Với mỗi thiết bị, đọc mốc thời gian đồng bộ gần nhất từ bảng sync_cursor để xác định điểm bắt đầu quét log mới.Bước 4: Kết nối vật lý tới thiết bị (ZKTeco SDK hoặc HTTP Digest Hikvision), thực hiện tải log chấm công phát sinh sau mốc cursor.Bước 5: Ghi toàn bộ log tải về vào database PostgreSQL.Bước 6: Nếu thành công, cập nhật con trỏ sync_cursor lên mốc thời gian của log mới nhất vừa tải để tối ưu lượng băng thông truyền tải cho chu kỳ quét tiếp theo.2.4. Thuật toán Khử trùng lặp Dữ liệu &amp; Giải thuật Tính công Ca đêm chuyên sâu2.4.1. Cơ chế Khử trùng lặp Log (Log Deduplication)Do hệ thống cho phép chạy song song cả cơ chế kéo chủ động (Pull) và thiết bị tự đẩy (ADMS Push), nguy cơ trùng lặp dữ liệu log chấm công là rất lớn. Để loại bỏ triệt để, hệ thống áp dụng cơ chế khử trùng lặp 2 lớp:Lớp 1 (Ràng buộc cứng DB): Thiết lập index độc nhất UNIQUE(device_id, employee_code, check_time) trên bảng attendance_log. Khi chèn dữ liệu, nếu trùng lặp sẽ bị cơ chế ON CONFLICT DO NOTHING của PostgreSQL loại bỏ trực tiếp.Lớp 2 (Kiểm tra phần mềm): Tầng Usecase trước khi chèn hàng loạt (Bulk Insert) sẽ đối chiếu với log hiện hữu trong khoảng thời gian scan để lọc bỏ bản ghi cũ trước khi gửi xuống DB.2.4.2. Giải thuật Tính công Ca đêm qua ngày (Overnight Shift Algorithm)Bài toán: Nhân viên ca đêm bắt đầu check-in từ 21:00 tối hôm trước và check-out lúc 06:00 sáng ngày hôm sau. Nếu phân tích theo ngày dương lịch thông thường, hệ thống sẽ ghi nhận ngày hôm trước thiếu giờ check-out và ngày hôm sau thiếu giờ check-in, dẫn đến tính sai công hoàn toàn.Giải thuật xử lý trong mã nguồn hệ thống:Bước 1: Đọc lịch gán ca (employee_shift) của nhân viên tại ngày bắt đầu quẹt thẻ (Ngày D). Xác định xem ca đó có phải là ca đêm hay không (giờ kết thúc nhỏ hơn giờ bắt đầu).Bước 2: Tìm lượt check-in đầu tiên (First In) của nhân viên trong ngày D nằm trong khung thời gian ân hạn bắt đầu ca (ví dụ: từ 19:00 đến 23:00).Bước 3: Thiết lập cửa sổ tìm kiếm check-out (Check-out Window) kéo dài từ giờ Check-In đến tối đa MaxWorkingMinutes của ca gán (ví dụ: ca đêm 8 tiếng, MaxWorkingMinutes là 840 phút = 14 tiếng). Cửa sổ tìm kiếm Check-Out sẽ kéo dài tới 11:00 trưa ngày hôm sau (Ngày D+1).Bước 4: Quét toàn bộ log chấm công thô của nhân viên trong cửa sổ tìm kiếm trên. Lượt quẹt cuối cùng tìm thấy sẽ được gán làm giờ Check-Out (Last Out) cho ngày công D.Bước 5: Tổng hợp số giờ làm việc thực tế = (Last Out - First In) - BreakMinutes. Khi nhấn bắt đầu, máy chấm công sẽ chuyển sang chế độ đăng ký, nhân viên đặt ngón tay 3 lần. Server tự động lưu trữ template Base64 vào database trung tâm.Sao chép &amp; Đồng bộ Thiết bị (Backup &amp; Sync Device): Hỗ trợ chọn thiết bị nguồn và thiết bị đích trống, sau đó sao chép toàn bộ danh sách nhân viên và mẫu vân tay đã lưu trữ sang thiết bị đích một cách tự động và an toàn.7.4. Phân hệ Quản lý Ca &amp; Gán ca (Shifts &amp; Assignment)Phân hệ thiết lập lịch làm việc và gán ca để làm cơ sở cho engine tính toán công ngày:Khai báo ca làm việc linh hoạt: Cho phép tạo mới các ca làm việc với các thông số: Tên ca (Ca Hành chính, Ca Sáng, Ca Chiều, Ca Đêm...), Giờ bắt đầu ca, Giờ kết thúc ca, Thời gian nghỉ giải lao (phút), Số phút đi muộn cho phép (Late Grace), Số phút về sớm cho phép (Early Grace) và Mã màu sắc (Color Code) để hiển thị lịch.Gán ca làm việc (Assign Shift): Cho phép gán ca làm việc cho nhân viên theo chu kỳ thời gian (Từ ngày... Đến ngày...). Hệ thống sẽ lưu vết lịch sử gán ca của từng nhân viên.Tính công tự động (Process Attendance): Giao diện cho phép HR chọn một ngày cụ thể và bấm nút 'Tính công'. Engine backend sẽ chạy quy trình đối chiếu toàn bộ log thô của ngày đó với lịch gán ca để tính ra giờ vào/ra đầu cuối, số phút trễ/sớm và tổng giờ công thực tế.Chuẩn hóa định dạng thời gian: Hệ thống tự động chuẩn hóa định dạng thời gian từ database (HH:MM:SS) và client (HH:MM) về định dạng chuẩn 'HH:MM' khi sửa ca làm việc, giúp giao diện hiển thị sạch sẽ và API xử lý nhất quán.7.5. Phân hệ Đơn từ &amp; Tự phục vụ (ESS - Employee Self Service)Cung cấp cổng thông tin tự phục vụ cho người lao động để giảm tải cho bộ phận nhân sự và minh bạch hóa dữ liệu:Nộp đơn đăng ký: Người lao động hoặc HR đại diện gửi các loại đơn từ gồm:  - Đơn nghỉ phép (Leave Request): Chọn loại phép (nghỉ phép năm, nghỉ ốm, thai sản, không lương), khoảng ngày nghỉ và lý do giải trình.  - Đơn làm thêm giờ (Overtime Request): Chọn ngày làm thêm, mốc giờ bắt đầu/kết thúc làm thêm và lý do tăng ca.  - Đơn giải trình sửa giờ công (Attendance Correction): Cho trường hợp quên quẹt thẻ/vân tay, chọn ngày cần sửa, loại check (Vào/Ra), mốc giờ thực tế muốn ghi nhận và lý do giải trình.Quản lý &amp; Phê duyệt đơn (Approval System): Giao diện lọc đơn theo loại và trạng thái (Chờ duyệt, Đã duyệt, Từ chối). HR/Admin có quyền xem xét lý do, bấm duyệt (Approve) đơn để hệ thống tự động cập nhật công (đối với đơn sửa công, hệ thống tự động sinh log ảo chấm công tương ứng) hoặc từ chối (Reject) đơn.Bộ phân tích ngày linh hoạt (Flexible Date Parsing): Cải tiến API tiếp nhận đơn từ ESS để tự động hỗ trợ nhiều định dạng ngày giờ khác nhau từ client gửi lên (như YYYY-MM-DD hay RFC3339 timestamp), giúp đảm bảo quá trình đăng ký luôn ổn định.7.6. Phân hệ Dữ liệu Chấm công (Attendance Data)Giao diện tra cứu dữ liệu chấm công chi tiết của nhân sự, hỗ trợ lọc theo mốc thời gian và mã nhân viên:Tóm tắt công ngày (Daily Attendance Tab): Hiển thị bảng tổng hợp công sau khi tính toán gồm: Tên nhân viên, Ngày làm việc, Ca áp dụng, Giờ Vào đầu tiên (First In), Giờ Ra cuối cùng (Last Out), Số phút đi muộn, Số phút về sớm, Số giờ làm việc thực tế, Số giờ tăng ca (OT), Ngày nghỉ phép liên đới và Trạng thái công ngày (Đi làm, Muộn, Vắng, Phép).Nhật ký log thô (Raw Logs Tab): Hiển thị danh sách tất cả các bản ghi chấm công thô thu thập từ các máy chấm công gồm: Thời gian check-time chính xác, Tên/Mã nhân viên, Loại check (Vào/Ra), Phương thức xác thực và Máy chấm công ghi nhận.Tính năng kiểm soát giờ quét ±1 tiếng (Validity Check): Cột 'Kiểm tra Giờ' trên bảng hiển thị trạng thái hợp lệ của mỗi lần quét:  - Nhãn 'Trong giờ' (Valid) màu xanh: Log quẹt nằm trong khoảng thời gian làm việc ca gán ±1 tiếng.  - Nhãn 'Ngoài giờ' (Invalid) màu đỏ: Log quẹt nằm ngoài khoảng thời gian làm việc ca gán ±1 tiếng (lý do: Outside shift window +-1h) hoặc nhân viên chưa được gán ca làm việc (lý do: No shift assigned). Toàn bộ log ngoài giờ này sẽ không được engine tính công tính vào giờ vào đầu/ra cuối.7.7. Phân hệ Báo cáo Tháng (Monthly Reports)Phân hệ tổng hợp bảng công tháng và xuất file báo cáo phục vụ tính lương:Bảng ma trận công tháng: Hiển thị danh sách nhân viên ở các hàng, các ngày trong tháng ở các cột. Mỗi ô ngày hiển thị ký hiệu trạng thái công (P: Đi làm đúng giờ, L: Đi muộn, A: Vắng mặt, V: Nghỉ phép).Tooltip tương tác: Khi di chuột (hover) vào từng ô ngày, hệ thống hiển thị chi tiết giờ Vào/Ra, số giờ làm việc, đi trễ, về sớm, và thống kê số lần quét vân tay hợp lệ / không hợp lệ của ngày đó.Thống kê quét vân tay tháng: Bảng báo cáo tích hợp 2 cột tổng kết ở cuối bảng:  - Quét hợp lệ (Valid Scans): Tổng số lần chấm công hợp lệ trong tháng.  - Không hợp lệ (Invalid Scans): Tổng số lần chấm công không hợp lệ (ngoài khung giờ hoặc không gán ca).Xuất Excel (CSV): Cho phép tải file báo cáo chấm công tháng chi tiết về máy tính để bộ phận kế toán tính lương.Gửi báo cáo qua Gmail: HR bấm nút gửi báo cáo tháng, hệ thống tự động tạo file CSV bảng công chi tiết cá nhân và gửi email trực tiếp tới hòm thư của từng nhân viên.7.8. Phân hệ Nhật ký Hệ thống &amp; Đồng bộ (Audit Logs &amp; Sync History)Phân hệ ghi nhận dấu vết hệ thống phục vụ mục tiêu kiểm toán bảo mật và giám sát đồng bộ:Nhật ký hoạt động (Audit Logs): Bảng ghi chi tiết các hành động quản trị hệ thống: Thời gian thực hiện, Tài khoản thực hiện, Loại hành động (CREATE, UPDATE, DELETE thiết bị/nhân viên), Mô tả chi tiết nội dung thay đổi và Địa chỉ IP của client gọi API.Lịch sử đồng bộ (Sync History): Thống kê chi tiết các phiên đồng bộ dữ liệu: Thiết bị thực hiện, Loại đồng bộ (nhân sự, chấm công), Hình thức kích hoạt (Thủ công/Tự động), Trạng thái (Thành công/Thất bại), Số bản ghi xử lý thành công, Thời gian và thông báo lỗi chi tiết nếu có.Đồng bộ ngay (Sync Now): Nút bấm ép buộc hệ thống chạy ngầm quét và tải log chấm công tức thời từ toàn bộ thiết bị đang online.7.9. Phân hệ Cấu hình hệ thống (Settings)Hiển thị các thông tin cấu hình cơ bản của hệ thống và bảng tài liệu hướng dẫn nhanh từng bước vận hành các chức năng cốt lõi cho quản trị viên mới.
